## 오늘 요약 (공유용)

어제 끝내 못 고친 지도 마커 클릭 문제를 마무리하는 걸로 하루를 시작했다. 그다음 참가자 핵심 플로우(사진 제출 → AI 판정 → 미션 공개 → 영상 업로드 → 인벤토리)와 임원 화면(도움 요청, 지급 확정, 대시보드, 영상 모아보기, 비밀 경로 setup)을 전부 이어붙여서 앱이 처음부터 끝까지 돌아가게 만들었고, 확정해둔 디자인 방향도 처음 입혔다.

오후엔 그 디자인을 다시 들여다봤다. 지도(Kakao)를 실제로 화면에 넣어보니 "종이/카키톤" 베이스가 지도 색이랑 겹쳐서 애매했고, 재검토 끝에 얇은 테두리 + 흰 카드 톤으로 정리했다. GPT한테도 같은 화면을 보여주고 교차 검토를 받았는데, "장난감 같다"는 지적을 받아 한 번 더 다듬었고, "Borderless"라는 이름 자체가 화면에 안 느껴진다는 지적도 받아 진행 스테퍼 + 연결선 모티프로 반영했다. 기능적으로는 미션 통과 후 영상 업로드를 별도 화면(탭)으로 분리했고, **영상 업로드까지 끝나야 다음 지역으로 넘어가도록** 규칙을 추가했다. Vercel 배포도 이번에 처음 성공시켰다 — 실기기(아이폰)로 테스트하면서 내 위치가 안 보이는 버그, 도움요청 버튼이 화면을 가리는 버그 등을 잡았고, 마지막엔 네이버지도처럼 폰이 보는 방향을 나침반으로 보여주는 기능도 추가했다.

## 디자인 결정 과정 (포트폴리오용 메모)

처음 확정했던 "오리엔티어링 컨트롤"(종이/카키 톤 + 두꺼운 잉크 테두리 + 원형 스탬프)은 지도를 넣기 전에 정한 방향이었다. 오전에 이 디자인을 실제 화면에 처음 입혔고, 오후에 Kakao 지도를 그 위에 앉혀보니 지도의 옅은 기본 스킨과 배경 톤이 명도상 잘 구분되지 않아 "지도 반 + 커스텀 UI 반"이 따로 노는 느낌이 들었다.

이후 진행한 순서:
1. 지도-패널 경계를 어떻게 처리할지 4안(계기판 구획선/통합 프레임/바텀시트/눈금자) 비교 → 통합 프레임 선택
2. 배경 팔레트를 지도 색과 나란히 4방향 비교 → 카키 페이지 위에 흰 계기 카드를 얹는 투톤 방식 선택. 이 화면을 GPT/Gemini/(별도) Claude 세 곳에 각각 보여주고 독립적으로 교차 검토받음 — 팔레트 방향은 거의 동일하게 수렴(서로 다른 AI가 같은 결론에 도달한 것으로 교차 검증됨), "지도-패널 위계가 약하다"는 지적은 유효했지만, 실제 화면에 없는 요소("네이비 헤더" 등)를 지적한 부분은 걸러냄
3. "Borderless"라는 이름 자체를 화면에 어떻게 표현할지 고민 — 보물지도(트레저맵) 느낌도 검토했지만, 그 스타일을 제대로 살리려면 진한 크래프트지 톤으로 가야 해서 이미 기각한 "여권/스탬프북형 크래프트지" 방향과 겹침 → 대신 "경계가 열리는 프레임"(평소엔 닫혀 있다가 다음 지역으로 넘어갈 때만 테두리 한 지점이 찢어지듯 열리고 점선 경로가 빠져나가는 연출)을 직접 제안해 시각화
4. 사용자 피드백: "내가 만든 디자인이 장난감 같고, GPT 게 더 어른스럽다" → 원인을 분석해보니 색이 아니라 **두꺼운 flat 테두리 / 알약형 배지·버튼 / 원형 고무도장 스탬프**가 범인이었음. 헤어라인 테두리 + 미세한 그림자 + 각진 permit 카드로 교체
5. 이 사이 사용자가 GPT 자체 기획안(진행바/프레임 확장/빛 리빌/Framer Motion+Pretendard 제안 등)을 공유 → 비판적으로 검토: "프레임이 진행률만큼 확장"안은 "모든 마커는 항상 노출돼야 한다"는 기존 요구사항과 충돌, "빛이 퍼지는 완료" 리빌은 이미 확정한 스탬프 리빌과 클라이맥스가 중복, Framer Motion/Pretendard 도입은 답사지의 불안정한 통신 환경을 고려하면 리스크, 제안된 세이지그린 보조색은 팀 B 컬러(녹색)와 혼동 우려 — 팔레트 수치는 우리 안과 수렴했다는 점만 채택
6. GPT 재검토: 어른스러운 톤은 잡았지만 "Borderless"라는 이름의 의미가 안 느껴진다는 지적 → 진행 스테퍼(a-b-c-d) + 현재 위치 점 + 연결선을 최소한으로 얹어 "경계를 넘어가는 여정"을 정보 구조로 표현. 파단선(CAD 표기법) 아이디어는 "일반 사용자가 의미를 못 읽는다"는 피드백으로 실제 연결선으로 대체

| 어드민 폼 테스트에서 값이 잘못 읽힘 | `querySelector`가 재료 폼의 name 입력을 먼저 매칭(같은 이름의 필드가 여러 폼에 존재) | 제출 버튼에서 `closest('form')`으로 스코프 좁혀서 조회 |
| 영상 업로드가 Vercel 요청 크기 제한에 걸릴 위험 | 영상 바이트가 우리 서버(API Route)를 그대로 통과하는 구조 | 서명된 업로드 URL 발급 → 브라우저에서 Supabase Storage로 직접 업로드하는 3단계 흐름으로 변경 |

| Vercel 빌드 실패 (Prisma client 없음) | 생성된 Prisma client(`src/generated/prisma`)가 gitignore돼 있고 `postinstall` 스크립트가 없어 재생성이 안 됨 | `package.json`에 `postinstall: "prisma generate"` 추가 |
| Vercel Import 화면에 저장소가 안 보임 | Vercel GitHub App이 다른 저장소에만 접근 권한을 받아둔 상태 | "Adjust GitHub App Permissions"에서 권한 추가 |
| Vercel 빌드: `Error: supabaseUrl is required` | 필요한 환경변수가 아직 등록 안 됨 | 9개 환경변수(DB/Supabase/OpenAI/Kakao/ADMIN_SECRET_PATH) 전부 등록 |

| 지도를 축소하면 내 위치 마커가 안 보임 | 실제 좌표축척 반경 15m `Circle`이라 줌아웃 시 사실상 소실 | 마커 핀처럼 고정 픽셀 크기 `CustomOverlay`로 교체, 색도 핀과 다른 마젠타로 구분 |
| 임원 도움 요청 버튼이 패널 텍스트를 가림 | 전역 `fixed` 버튼과 패널 하단 콘텐츠 영역이 겹침 | 패널 하단에 여유 패딩(`pb-16`) 확보 |
| "다음 목적지: A지역" 문구가 이미 A에서 작업 중인데 표시됨 | 영상 업로드 전엔 지역이 "완료"로 안 잡히는데, 문구는 그대로 "다음 목적지"로 고정돼 있었음 | 영상 대기 중이면 "A지역 · 영상 업로드 남음"으로 문구 분기 |
| 영상 업로드 후에도 다음 지역으로 안 넘어감 | 지역 진행 상태는 서버 컴포넌트가 페이지 로드 시 한 번만 계산해 내려주는 값이라, 클라이언트 상태 변경만으론 안 갱신됨 | 사진 제출/영상 업로드 성공 직후 `router.refresh()` 호출 |
| 나침반 부채꼴이 안 움직임 | `"ondeviceorientationabsolute" in window` 특성 감지가 일부 브라우저에서 오탐(그 이벤트가 실제론 안 터지는데도 존재하는 것처럼 감지됨) | 두 이벤트(`deviceorientationabsolute`/`deviceorientation`) 모두 구독하고, 데이터 안에 신뢰 가능한 값(`webkitCompassHeading` 또는 `absolute===true`)이 있을 때만 반영 |
| 나침반 켜기 버튼이 눈에 안 띔 | 흰색 알약형 배경이라 배경과 대비가 약함 | 다른 CTA와 같은 오렌지 액센트 스타일 + 이모지로 교체 |
| "사진 찍기"/"영상 촬영" 문구가 실제 동작과 안 맞음 | `<input capture="environment">`가 기기·브라우저에 따라 카메라 대신 갤러리 선택창을 열기도 함(강제 촬영이 아님) | 문구를 "사진 선택"/"영상 선택"으로 변경 |
| 사진 제출도 영상과 같은 이유로 Vercel 요청 크기 제한 위험 | 사진 바이트가 우리 서버(API Route)를 그대로 통과하는 구조 (AI 판정은 이미 URL만 사용) | 서명된 업로드 URL 발급 → 브라우저에서 Supabase Storage로 직접 업로드하도록 변경 |

## 느낀 점 / 메모

- **여러 AI에게 교차 검토를 받으니 진짜 정확한 지적이 나왔다.** "색은 맞는데 형태(테두리/모서리/스탬프)가 문제"라는 진단은 한쪽 의견만 들었으면 못 잡았을 것.
- **"이미 확정했다"고 고정된 규칙처럼 취급하면 안 됐다.** 지도를 실제로 넣기 전에 정한 팔레트였는데, 그 전제가 바뀌었으면 다시 열어봐야 했다 — CLAUDE.md도 매번 재검토 대상.
- **AI(나)가 자기가 만든 결과물을 방어하려는 편향이 있었다.** "이건 이미 확정한 거라 안 된다"며 GPT 제안을 방어적으로 반박하다가, "너도 열린 생각을 가져야 한다"는 지적을 받고서야 CLAUDE.md 자체도 재검토 대상이라는 걸 인정했다. 앞으로는 내가 이전에 내린 결정이라고 자동으로 방어하지 말 것.
- **로컬에서 고친 걸 커밋·푸시하지 않고 "완료"라고 보고한 적이 있었다.** 내 위치 마커 수정을 로컬에만 만들어두고 실기기로 검증해보라고 했는데, 실제 배포본엔 예전 버그가 그대로 있었다. 코드 수정 → 배포 확인을 요청하기 전에 반드시 커밋·푸시 여부를 먼저 확인해야 한다.
- **배포 트러블슈팅은 거의 다 "설정" 문제였지 코드 문제가 아니었다.** 환경변수 Sensitive, 리전 불일치, 도메인 미등록 — 로컬에서 완벽해도 플랫폼 설정에서 막힐 수 있다는 걸 실감.
- **서버 컴포넌트 props와 클라이언트 상태가 분리돼 있다는 걸 깜빡해서 버그를 만들었다.** 영상 업로드 후 진행이 안 되던 문제 — `router.refresh()`가 필요한 지점을 설계 단계에서 미리 챙겼어야 했다.
- **오래 켜둔 dev 서버가 스키마 변경을 못 따라갈 수 있다.** 마이그레이션 이후엔 그냥 재시작하는 걸 습관화하는 게 디버깅 시간을 아꼈다.
- **소스 오브 트루스 문서(기획서)가 실제 구현과 어긋난 채로 방치돼 있었다.** 이미 폐기한 기능(재료 교환)과 뒤바뀐 결정(임원 로그인)이 원본 문서에 그대로 남아 있었음 — 큰 변경 후에는 관련 문서도 같이 훑어보는 걸 루틴화해야.
