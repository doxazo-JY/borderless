# 데일리 노트 — 2026.07.20

## 오늘 요약 (공유용)

어제(7/19)까지의 버그 수정·기능 추가 이후, 오늘은 배포 파이프라인 점검과 어드민 쪽 새 포인트 등록 흐름 개선이 중심이었다. 먼저 `ADMIN_SECRET_PATH` 환경변수를 바꾼 뒤 재배포가 제대로 반영이 안 되는 것처럼 보이는 문제를 같이 조사했는데, 실제로는 Vercel 배포 자체는 계속 성공하고 있었고 반영이 좀 늦게 된 것으로 확인됐다(시간이 지나니 해결). 이어서 답사 예정지 근처에서 실제로 찍은 사진 3장(아파트 정문, emart 사거리, 상가 외벽 간판)을 놓고 AI 판정용 기준 사진으로 쓸 만한지 검토하고, 각각에 맞는 판정 프롬프트(judgePrompt) 3개를 작성했다. 마지막으로 어드민 "새 포인트 추가" 폼을 개선했다 — 지도를 클릭해서 좌표를 직접 찍을 수 있게 됐고, 미션 항목도 삭제/추가만 되던 걸 인라인으로 수정할 수 있게 됐다.

세션 막판에 버그 3건이 추가로 보고됐는데, 이 노트를 쓰는 시점까지는 아직 코드 수정 전이다 — 아래 "다음에 할 일" 참고.

## 오늘 한 일

### 1. Vercel 재배포 트러블슈팅

- 증상: `ADMIN_SECRET_PATH` 환경변수를 Vercel 대시보드에서 바꾸고 재배포했는데 `/admin/{secret}/setup`이 계속 404
- Vercel 상태 페이지(vercel-status.com) 확인 → 그 날짜엔 전체 시스템 정상, 플랫폼 장애 아님
- 배포 목록에서 "Initializing"에 멈춘 것처럼 보이는 배포가 있었지만, 실제 요청 로그(Logs 탭)를 보니 그 사이에도 다른 배포들은 정상적으로 200을 응답하고 있었음 — 즉 배포 자체는 계속 성공 중이었고, "Initializing" 표시가 실제 상태를 정확히 반영 못 한 것으로 추정
- 진짜 원인은 새로 설정한 env var 값이 아직 프로덕션에 반영된 배포로 승격(promote)되지 않은 상태였던 것으로 보임. Vercel 대시보드 Settings → Environment Variables에서 `ADMIN_SECRET_PATH`가 Production 환경에 정확한 값으로 등록됐는지, Deployments에서 최신 배포가 실제로 Production에 연결(promote)됐는지 확인하는 절차를 거침
- 결론적으로는 별다른 코드 수정 없이 시간이 좀 지나니 해결됨 (반영 지연으로 추정, 재현되면 다시 조사 필요)
- 트리거 겸 사소한 문서 동기화로 커밋 `592544b` (`CLAUDE.md` 상태 문단을 7/19 시점 실제 작업 내용으로 갱신) 푸시

### 2. 실사진 기반 판정 테스트 준비

사용자가 답사 예정 지역 근처에서 실제로 찍은 사진 3장을 검토했다:
- 아파트 정문 (아치형 구조물, "2차 아파트" 간판, 노란 현수막)
- emart 사거리 (건물 로고, 옥상 큐브 간판, 신호등 트러스 구조물)
- 상가 외벽 (다이소/TOPTEN10/S-market 간판)

각 사진을 기준 사진으로 썼을 때 참가자가 다른 각도/배율로 찍어도 같은 장소로 인식되도록, 랜드마크 요소를 나열하고 "2개 이상 겹치면 통과"로 판단하게 하는 judgePrompt를 3개 작성해서 전달함 (실제 어드민 등록·사진 업로드는 사용자가 직접 진행 예정 — 코드 변경 없음).

### 3. 어드민 "새 포인트 추가" 폼 — 지도 클릭 좌표 지정

- 새 파일 [src/lib/kakao-loader.ts](../src/lib/kakao-loader.ts) — Kakao Maps SDK(services 포함) 지연 로드 함수를 `LocationForm.tsx`에서 분리해서 공용화
- 새 파일 [src/components/admin/LocationMapPicker.tsx](../src/components/admin/LocationMapPicker.tsx) — 작은 지도 위젯. 클릭하면 그 좌표로 마커 이동 + 위/경도 필드 자동 채움. 기존 등록된 16개 포인트는 회색 점으로 같이 표시(참고용)
- [src/components/admin/LocationForm.tsx](../src/components/admin/LocationForm.tsx) 흐름 재구성: 기존엔 "주소 검색 / GPS 버튼 / 좌표 직접입력 / (없던) 지도"가 서로 분리된 느낌이었는데, 이제 **주소 검색 → 지도가 그 근처로 자동 이동+확대(레벨 3) → 지도 클릭으로 세부 위치 확정 → 좌표 필드는 확인/미세조정용**으로 순서가 이어짐. 주소·GPS로 인한 "큰 점프"일 때만 줌을 다시 당기고, 지도 클릭으로 인한 세부 조정은 관리자가 맞춰둔 줌을 그대로 유지
- [src/app/admin/[secret]/setup/page.tsx](../src/app/admin/%5Bsecret%5D/setup/page.tsx)에서 기존 16개 포인트 좌표를 `existingLocations`로 전달

## 다음에 할 일 (아직 미해결)

세션 막판에 아래 3건이 추가로 보고됨. 이 노트 작성 시점 기준 코드 수정 전이니, 다음에 이어서 처리할 것:

1. **새 포인트 추가 후 사진 미리보기가 초기화 안 됨** — `LocationForm`에서 포인트를 하나 추가하고 나면, 다음 포인트를 추가하려 할 때 이전에 선택했던 기준 사진 미리보기가 그대로 남아있음. `createLocation` 성공 후 폼(특히 `photoPreview` state와 파일 input)을 초기화해야 함.
2. **강화도 더미 포인트를 삭제 없이 "비활성화"할 수 있게 해달라는 요청** — 지금 등록된 16개 포인트(AI로 생성한 이미지 기반 강화도 더미)는 계속 테스트용으로 남겨두고 싶지만, 협업자가 자기 동네 근처에 새 포인트를 넣어서 직접 테스트해볼 수 있도록 강화도 포인트들은 잠깐 숨기고 싶어함. `Location`에 `isActive`(가칭) 같은 boolean을 추가해서, 비활성 포인트는 참가자 지도에서 제외하되 어드민에서는 계속 보이고 다시 켤 수 있게 하는 방향이 맞아 보임. 스키마 변경 필요.
3. **나침반 방향 표시 켜기 버튼을 매번 눌러야 하는 문제** — iOS Safari는 `DeviceOrientationEvent.requestPermission()`이 매 페이지 로드마다 사용자 제스처를 새로 요구하는 게 표준 동작이라, 브라우저 자체 정책일 가능성이 높음(웹 표준상 영구 허용을 저장할 방법이 없음 — 네이티브 앱과 다른 부분). 완전히 없앨 수는 없어도, 한 세션(탭을 안 닫는 동안) 안에서는 다시 안 눌러도 되게 할 수 있는지, 또는 최소한 버튼을 덜 거슬리게 만들 수 있는지 검토 필요. `src/components/KakaoMap.tsx`의 `requestCompassPermission`/`needsCompassPermission` 부분 참고.
