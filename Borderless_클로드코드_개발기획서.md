# Borderless 수련회 미션 프로그램 — 개발 기획서 (Claude Code용)

## 0. 프로젝트 한 줄 요약

교회 청년부 야외 수련회에서 사용할 웹앱. 참가자(그룹)가 지도/주소로 안내된 장소로 이동해 사진을 찍어 업로드하면, AI가 기준 사진과 비교해 통과 여부를 판정하고, 통과 시 해당 장소의 미션(말씀/찬양/기도/퀴즈)을 공개한다. 미션 완료는 대부분 수행 영상 업로드로 확정되지만, 퀴즈(PUZZLE) 타입은 예외로 정답 텍스트 제출로 확정된다. 재료 지급 자체는 임원이 현장에서 오프라인으로 확인 후 관리자 화면에서 확정 처리한다. 모바일 브라우저 기준으로 동작해야 한다.

---

## 1. 사용자 및 규모

| 항목 | 내용 |
|---|---|
| 참가자 | 총 16명, 4팀(A~D, 팀당 4명), 팀 내 2인 1조로 8그룹 운영 |
| 운영자(임원) | 4명 — 각자도 미션에 함께 참여하므로 실시간 상시 모니터링 불가 |
| 동시 사용자 | 최대 8그룹 (동시 접속 부하는 매우 낮은 수준) |
| 미션 장소 수 | 총 16곳 = **4개 지역(a/b/c/d) × 지역당 4개 방문포인트** (확정) |
| 지역별 필수 방문 | 각 그룹은 4개 지역(a/b/c/d) 중 **지역마다 방문포인트 1곳씩, 총 4곳을 반드시 방문**해야 함 (최소 방문 장소 수 하한선 = 4) |
| 장소별 재료 획득 캡 | **선착순 2그룹까지만** 해당 포인트의 재료를 획득 가능 (8그룹 기준). 지역당 4포인트×2그룹 = 8그룹 수용량으로, 8그룹 전원이 각 지역을 성공적으로 통과할 수 있는 구조 |
| 사용 환경 | 모바일 브라우저 (참가자), 데스크탑/모바일 겸용 관리자 화면 (임원) |
| 행사 시간 | 오전 약 2시간~2시간 20분 (미션 진행 구간) |
| 행사 일정 | 2026-08-16 ~ 2026-08-18 |
| 미션 장소 권역 | 숙소(인천 강화군 송해면 오류내길99번길 40-7)에서 약 30분 거리 반경 내에서 16개 포인트 선정 예정 |

> ⚠️ **네이밍 주의**: 팀은 대문자(A/B/C/D), 지역은 소문자(a/b/c/d)로 겹치게 표기되어 있어 헷갈리기 쉽다. 실제 UI 문구에서는 "A팀" / "a지역"처럼 구분하거나, 지역을 "1구역~4구역" 등으로 바꿔 표기하는 걸 권장. 데이터 모델상으로는 Team과 Region을 완전히 별개 엔티티로 분리한다.

## 2. 핵심 사용자 흐름 (참가자)

1. 그룹은 **소속 팀(A/B/C/D) + 조(1/2)를 선택**하는 방식으로 식별한다. 별도 로그인 코드나 QR 없이 목록에서 선택하는 정도로 충분하다. (팀/조 개수는 관리자가 설정 가능해야 함)
2. 앱 접속 후 **사이트에 내장된 실제 지도**(Kakao Maps 등 지도 API)에서 4개 지역(a/b/c/d)과 지역별 방문포인트를 확인한다.
   - 지도 앱과 우리 사이트를 오가지 않도록, 위치 확인부터 사진 업로드까지 전부 한 사이트 안에서 끝난다.
   - 지도에는 **브라우저 위치 권한을 받아 참가자의 현재 위치(파란 점)를 실제 지도 위에 함께 표시**한다. 순수 안내용 정적 이미지가 아니라 실좌표 기반 지도다.
   - 장소 자체(위치)는 공개되어 있다. 단, 그 포인트에서 어떤 재료를 주는지도, **선착순 잔여 캡 상태도 참가자에게는 절대 미리 공개하지 않는다.** 마감 여부는 실제로 가서 사진을 찍어봐야만 알 수 있다 (아래 5번 참고). 지도/목록 화면에는 우리 그룹이 이미 완료한 지역인지 정도만 표시한다.
   - GPS는 **판정/검증용이 아니라 순수 내비게이션 편의용**이다. 사진 판정 자체는 위치와 무관하게 AI가 이미지만으로 수행한다 (4절 참고).
   - 지역별로 4개 포인트가 있고, 그룹은 **각 지역마다 아무 포인트나 1곳을 골라 통과하면 그 지역은 완료**된 것으로 간주한다. 같은 지역의 다른 포인트를 중복 방문할 필요는 없다.
3. 원하는 장소를 선택하면, 그 장소의 **기준 사진**(찾아야 할 목표 오브젝트/풍경)이 화면에 표시된다.
4. 참가자는 해당 장소로 이동하거나(또는 이동 중 유사한 풍경 발견 시 그 자리에서) 사진을 촬영해 업로드한다.
5. 사진을 업로드하면 서버가 **먼저 그 포인트의 잔여 캡을 확인**한다. 이미 선착순 2그룹이 채워졌다면, AI 판정 여부와 무관하게 즉시 **"이미 마감된 포인트입니다"**로 응답한다 — 참가자는 이때 처음으로 마감 사실을 알게 된다. (같은 지역의 다른 포인트로 가서 다시 시도할 수 있다.)
6. 캡이 남아 있다면 그제서야 AI가 기준 사진과 업로드 사진을 비교해 통과 여부를 판정한다.
7. 통과 시:
   - **그 즉시 서버가 캡 1자리를 원자적으로 차감/예약한다** (동시에 여러 그룹이 통과해도 정확히 2그룹까지만 잡히도록 트랜잭션 처리 — 7절 참고). 이 예약 시점이 곧 "선착순 확정" 시점이다.
   - 해당 장소의 미션(말씀 묵상 / 찬양 / 기도 중 하나)이 화면에 공개된다.
   - 미션 수행 후, 그 장면을 10~20초 내외 영상으로 촬영해 **앱에 업로드**한다. (이 업로드는 통과/실패를 AI나 앱이 자동 판정하지 않는다 — 다만 **이 업로드가 끝나야 그 지역이 "완료"로 확정되고 다음 지역으로 넘어갈 수 있다.** 2026-07-15 확정: 사진 판정 통과만으로는 지역이 완료되지 않음.)
   - **재료의 물리적 지급(임원이 현장에서 실제 재료를 건네주는 것)은 오프라인으로 이뤄진다.** 임원이 미션 수행을 확인한 뒤 관리자 화면에서 "지급 확정" 처리를 하는데, 이 시점엔 캡은 이미 위 단계에서 확정돼 있으므로 이 버튼은 캡을 가르는 게이트가 아니라 "실제로 건네줬다"는 기록/체크 역할만 한다.
8. 실패 시(캡은 남아 있지만 사진이 기준과 다른 경우): **재시도 횟수 제한 없음** (사진을 다시 찍어 업로드). "임원에게 도움 요청" 버튼은 처음부터 항상 노출되어 있으며, 실패가 2~3회 이상 누적되면 넛지(강조 표시)로 더 눈에 띄게 안내한다. 버튼을 누르면 임원에게 알림이 가고, 임원이 수동으로 통과 처리할 수 있다.
9. **그룹은 정확히 4곳만 방문한다** — 4개 지역 각 1포인트씩, 그 이상도 그 이하도 아니다. 필수 4곳을 채우면 미션 진행은 종료된다 (추가 방문/추가 재료 획득 없음).

> **범위 제외 확정 (2026-07-14)**: 재료 교환 기능(원래 10번 항목)은 이번 버전에서 완전히 제외한다. 재료 교환은 오프라인에서 참가자/임원이 알아서 처리하고, 앱은 관여하지 않는다.

## 3. 핵심 사용자 흐름 (임원/관리자)

1. **사전 준비(행사 전)**
   - 팀/조 구조 설정 (팀 개수, 팀당 조 개수)
   - **지역(a~d) 등록 및 지역별 방문포인트(장소) 배정** — 지역당 4포인트 구조
   - 장소(포인트)별 실제 좌표 등록 (지도 API에 표시할 위치)
   - 장소별 기준 사진 업로드
   - 장소별 미션 콘텐츠(말씀 본문, 기도 주제 등) 입력 — 찬양은 자유곡이라 별도 텍스트 불필요
   - 장소별 재료 목록 입력 (기본재료/부재료 구분, 같은 재료의 버전 다양화 포함 — 예: 일반떡/치즈떡)
   - 장소별 획득 캡(기본값 2그룹, 장소별로 조정 가능하도록 설계)
   - 판정 프롬프트(질문 문구) 및 임계값 테스트/조정

2. **행사 중**
   - 실시간 모니터링은 필수 아님 (임원도 미션에 참여하므로)
   - 특정 그룹이 반복 실패해 "수동 확인 요청"을 보내면, 알림을 받고 승인/거절 처리 가능 (모바일에서 처리 가능해야 함)
   - **캡(선착순 2그룹) 마감 여부는 사진 업로드 시점에 서버가 자동으로 판단한다 — 임원이 수동으로 캡을 관리하거나 막을 필요는 없다.** 그룹이 미션 수행 영상을 업로드하고 현장에서 임원에게 완료를 보이면, 임원은 관리자 화면에서 해당 그룹·장소의 "지급 확정" 버튼을 눌러 **실제로 재료를 건네줬다는 것만 기록**한다 (캡은 이미 앱이 확정해뒀으므로 이 버튼이 지급을 막을 일은 없다).
   - 전체 그룹의 진행 상황(방문 장소 수, 획득 재료, 장소별 잔여 캡 — **임원 화면에서만 보이고 참가자에게는 비공개**)을 한눈에 보는 대시보드 (선택사항이지만 캡 관리 때문에 우선순위 높음)

3. **행사 후**
   - 제출된 미션 인증 영상 전체를 한 번에 확인/다운로드 (결산 영상 제작용)

## 4. AI 판정 기능 요구사항

### 4-1. 판정 방식
- **CNN 특징 비교, CLIP 임베딩 거리, MSE/PSNR/SSIM/FID 같은 픽셀·통계 기반 지표는 사용하지 않는다.**
- **Vision-Language 모델에게 자연어로 질문하는 방식(VQA)을 사용한다.**
  - 예: "기준 사진에는 저수지와 벤치가 함께 보입니다. 아래 업로드된 사진에도 저수지(또는 유사한 형태의 물 고인 공간)와 앉을 수 있는 구조물이 함께 보입니까?"
  - 모델은 예/아니오(+근거)로 응답하며, 이를 통과/실패로 매핑한다.
- 표지판 텍스트가 있는 장소는 OCR 성격의 질문(예: "이 사진 속 표지판에 'OOO'라는 문구가 보이는가?")으로 판정 가능. 이 경우 정확도가 더 높으므로 "다른 곳에서 발견 인정" 없이 정확 매칭용으로 사용 권장.
- 이 판정은 **사진(장소 도착 확인)에만 적용**되며, 미션 수행 영상은 AI 판정 대상이 아니다 (임원이 오프라인으로 확인).

### 4-2. API 연동
- OpenAI API(비전 지원 모델, GPT-4o 계열 — 정확한 모델명은 env var로 설정해 교체 용이하게)를 사용해 이미지 2장(기준 사진 + 업로드 사진) + 질문 프롬프트를 함께 전송. (2026-07-14: 원래 Claude API 검토했으나 결제 방식 이유로 OpenAI로 전환)
- 별도의 모델 학습(fine-tuning)이나 GPU 서버 구축 불필요. API 호출만으로 처리.
- 서버는 이미지 업로드를 받아 API에 중계하고, 응답을 파싱해 통과/실패를 반환하는 얇은 역할만 수행.

### 4-3. 판정 안정성 확보 방안
- 임계값/프롬프트는 사전 답사 사진으로 미리 테스트하되, 완벽한 사전 튜닝은 불가능함을 전제로 설계.
- 재시도 횟수 제한 없음.
- 임원 수동 승인("도움 요청") 경로는 처음부터 상시 제공, 반복 실패 시 넛지로 강조 (필수 기능).
- 인터넷 연결 불안정 상황을 고려해, 업로드 실패/타임아웃 시 재시도 안내 UI 필요.
- **캡(선착순 2그룹) 확인이 AI 판정보다 먼저 실행된다.** 이미 마감된 포인트라면 AI API를 아예 호출하지 않고 즉시 "마감" 응답을 준다 — 참가자 경험뿐 아니라 불필요한 API 비용도 줄어든다.

## 5. 데이터 모델 (초안)

- **Region (지역)**: id, name(a/b/c/d), 소속 location(포인트) 목록 (지역당 4개)
- **Location (장소/방문포인트)**: id, region_id, name, **실제 위도/경도 좌표**(지도 API 표시용), 기준 사진 URL, 판정 프롬프트, 연결된 mission_id, 연결된 ingredient_id 목록, **capacity(기본 2)**, **claimed_count**, **claimed_by(그룹 id 목록)**, **is_active(기본 true)** — false면 참가자 지도/지역 진행 계산/캡 확인/사진 제출 처리에서 전부 제외, 어드민에서는 계속 보이고 다시 켤 수 있음 (2026-07-20 추가)
- **Mission (미션)**: id, type(말씀/찬양/기도/퀴즈), content(본문/기도 주제/퀴즈 내용, 찬양은 비어있어도 됨), answer(퀴즈 타입 전용, 쉼표로 여러 정답 허용) — 퀴즈(PUZZLE) 타입은 다른 셋과 달리 완료 조건이 영상 업로드가 아니라 정답 텍스트 제출(공백/대소문자 무시하고 비교)이다 (2026-07-21 추가)
- **Ingredient (재료)**: id, name — 표시/변형(예: "떡(치즈떡)")도 name 하나에 다 담음, category/variant/is_base는 어느 로직도 참조하지 않는 표시 전용 필드였던 걸 확인하고 2026-07-21에 제거
- **Team (팀)**: id, name(A/B/C/D)
- **Group (그룹)**: id, team_id, 조 번호(1/2), member 정보, 방문 완료한 location 목록, 보유 ingredient 목록(그룹 간 교환 반영), 제출한 인증 영상 목록
- **SubmissionLog (제출 기록)**: group_id, location_id, 업로드 사진, **cap_status(제출 시점 캡 확인 결과: 가능/마감 — 마감이면 AI 호출 자체를 생략)**, AI 판정 결과(통과/실패/사유, cap_status가 마감이면 null), 제출 영상 URL, **grant_status(대기/지급확정)**, **confirmed_by(임원 id)**, **confirmed_at**, timestamp — 결산 영상 제작 및 사후 검수용

> **핵심 제약 규칙 (변경됨)**: 캡 차감은 "지급 확정" 시점이 아니라 **AI 판정 통과 시점**에 일어난다. 서버는 사진 업로드 요청이 들어올 때마다 ①해당 location의 claimed_count < capacity(2), ②해당 group이 그 location의 region에서 아직 캡을 잡은 적 없음, 두 조건을 먼저 검증하고, 통과 시에만 AI 판정을 진행한 뒤 AI가 pass를 반환하는 즉시 claimed_count를 원자적으로 +1 한다. 여러 그룹이 동시에 마지막 한 자리를 두고 경합할 수 있으므로 이 증가 연산은 DB 트랜잭션/조건부 업데이트(예: `UPDATE ... SET claimed_count = claimed_count + 1 WHERE claimed_count < capacity`)로 원자적으로 처리해야 한다 (7절 참고). "지급 확정"은 이미 확정된 캡을 실제로 건네줬다는 기록일 뿐, 캡을 재검증하지 않는다.

## 7. 비기능 요구사항

- **모바일 우선(Mobile-first)**: 참가자는 전원 모바일 브라우저로 접속. 카메라 촬영 → 즉시 업로드 플로우가 매끄러워야 함.
- **네트워크 취약 환경 고려**: 야외 촬영 장소는 통신 상태가 불안정할 수 있음. 업로드 실패 시 재시도, 로딩 상태 명확히 표시.
- **배포 환경**: 별도 GPU 서버 불필요. 가벼운 웹 서버(백엔드) + AI API 호출 구조로 무료/저비용 클라우드 호스팅 사용 가능 (예: Render, Railway, Fly.io 등 — 최종 선택은 개발 시 결정).
- **비용**: AI API 호출은 8그룹 × 여러 장소 시도 기준으로 소규모 트래픽이라 비용 부담이 크지 않을 것으로 예상. 실제 호출 횟수 기준으로 사전 견적 확인 권장.
- **행사 당일 안정성**: 리허설(전날 또는 당일 아침) 시 실제 조명 조건으로 재검증 필요.
- **캡 동시성 (중요)**: 캡 차감이 참가자 앱에서 사진 업로드 즉시 자동으로 일어나므로, 같은 포인트의 마지막 자리를 두고 여러 그룹이 실제로 동시에 경합할 수 있다. "지급 확정" 단계(임원의 수동 조작이라 경합 위험 낮음)와 달리 이 부분은 반드시 서버 트랜잭션/조건부 업데이트로 원자적 처리해야 한다. 그렇지 않으면 2자리인데 3그룹이 통과 처리되는 오버부킹이 생길 수 있다.
- **지도 API 연동**: **Kakao Maps JavaScript SDK 사용 (무료)**. 카카오 개발자 계정당 첫 번째로 활성화한 앱은 하루 30만 건 무료 쿼터가 제공되어, 이 행사(8그룹, 반나절)규모에서는 비용이 사실상 발생하지 않는다. 사이트에 임베드해 실좌표 기반 지도 + 마커를 표시하고, 브라우저 `navigator.geolocation`으로 참가자 현재 위치(파란 점)를 지도에 함께 표시하므로, HTTPS 배포와 위치 권한 요청 UX가 필요하다. 위치 권한을 거부한 경우의 대체 동작(예: 마커만 보여주고 내 위치는 생략)도 고려해야 함.

## 8. 아직 확정되지 않아 개발 중 유연하게 처리해야 할 사항

- 판정 임계값/프롬프트 최종본 (답사 사진 확보 후 테스트하며 조정)
- 팀/그룹 간 재료 교환 규칙 (완전 자유 vs 일부 가이드)

### 8-1. 캡/지역 수용량 관련 확정 사항 (결론)
- **모든 포인트의 캡은 예외 없이 2로 통일한다.** 지역당 4포인트×캡2=8그룹 수용량이 정확히 그룹 수(8)와 맞아떨어지는 구조라, 포인트마다 캡을 다르게 하면 이 산수가 깨진다.
- **그룹당 같은 지역에서의 지급은 1회로 서버에서 강제 제한한다** (이미 지급받은 지역의 다른 포인트에서 중복 지급 불가, 2026-07-15 구현 완료 — 재시도하면 "이미 이 지역 OO포인트에서 통과했다"는 안내와 함께 서버에서 차단). 이 규칙만 지키면 "지역 전체가 소진됐는데 그 지역을 못 넘은 그룹이 남는" 상황은 산수상 발생하지 않는다. 유일하게 남는 리스크는 "자리 부족"이 아니라 "시간이 부족해서 그 지역 어떤 포인트도 시도조차 못 해본 그룹"인데, 이는 기존 임원 도움 요청 버튼으로 커버된다.
- **지역 "완료" 시점은 사진 판정 통과가 아니라 미션 영상 업로드까지 끝난 시점이다** (2026-07-15 확정). 그 전까지는 그룹이 그 지역에 계속 "머물러 있는" 상태로 취급되고, 지도 상단의 다음 목적지 안내도 그 지역을 계속 가리킨다.

### 8-2. GPS/위치 관련 범위 정리
- **포함**: 사이트 내장 지도 API에 실좌표 마커 표시 + 참가자 현재 위치(파란 점) 표시. 순수 내비게이션 편의 목적.
- **포함 안 함**: 사진 업로드 시 GPS 반경으로 "진짜 그 장소에서 찍었는지" 검증하는 로직은 지금은 구현하지 않음 (신뢰 기반). 필요시 추가할 수 있도록 Location에 좌표 필드는 이미 반영되어 있음.

## 9. Claude Code 진행 시 제안 순서

1. 데이터 모델 및 기본 백엔드(팀/조/지역/장소/미션/재료/캡 CRUD) 구축

5. 관리자 화면 구현 (지역/장소/미션/재료/캡 등록, 수동 승인, 지급 확정 처리)

7. 모바일 환경 실기기 테스트 (카메라 업로드, 지도/위치 권한, 네트워크 지연 상황 포함)
8. 배포 및 최종 리허설
